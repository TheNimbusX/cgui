# Кабинет клиента — Спецификация

## 1. Что мы создаём

Кабинет клиента — это веб-приложение, в котором клиент отслеживает свои заказы у
компании — заявки, счета, доставки — и с одного взгляда видит, на каком этапе
каждый из них и сколько осталось оплатить.

Клиент — это специалист по закупкам в компании, которая покупает промышленное
оборудование: станки, автоматику, комплектующие для производства. Он создаёт
заявку, получает коммерческое предложение и счёт, оплачивает и получает товар.
Кабинет показывает этот путь.

---

## 2. Видение продукта

Это согласованный замысел продукта. Конкретный UX — на усмотрение разработчика;
этот раздел определяет дух и бизнес-смысл.

**Основная идея.** Это трекер статусов в духе Wise или PayPal, а не часть
тяжёлой CRM. Ментальная модель клиента — «я отправил заявку — где она и сколько
я должен?». Кабинет отвечает на это с одного взгляда.

1. **Заявка — единственная сущность для навигации.** Счёт, заказ, доставка и
   УПД не имеют для клиента самостоятельного смысла — они существуют внутри
   заявки. Поэтому есть один список заявок, и всё остальное открывается изнутри
   заявки.

2. **Главная — это дашборд.** У большинства клиентов 2–3 активные заявки.
   Страница дашборда может закрывать 90% потребностей клиентов. Карточка заявки
   может показывать: что было заказано (кратко), трекер статусов, деньги
   (выставлено / оплачено) и следующий ожидаемый шаг.

3. **Заявка открывается во всех деталях** — все её данные на одном экране.

4. **Статус — это индикатор прогресса.** Клиент видит несколько ключевых
   статусов как пошаговый трекер — как отслеживание посылки или перевода Wise.
   Он всегда видит, где находится заявка и что будет дальше.

5. **Деньги — это то, что в счёте.** Главные цифры — итог счёта и оплаченная
   сумма. Бухгалтерская детализация скрыта за элементом «показать больше»
   (постепенное раскрытие).

6. **Список заявок.** Список заявок показывает все активные заявки. Пользователь
   может переключить вкладку, чтобы увидеть завершённые и отменённые заявки.

**Итого — три экрана:** главная (дашборд, реквизиты, сводные виджеты), список
заявок, страница заявки. Простой язык, без юридического жаргона;

## 3. Глоссарий

| Русский термин | Английский термин | Значение |
|---|---|---|
| Заявка | Request | Заявление клиента о потребности (товары, желаемая цена). Единственная сущность, по которой навигируется клиент. |
| КП | Proposal | Коммерческое предложение, отправленное клиенту. Структурно — проекция счёта. |
| Счёт | Invoice | Счёт, выставленный клиенту; клиент вносит по нему платежи. |
| Заказ | Order | Закупка товара компанией. Клиент видит даты заказа и статус груза. |
| Доставка | Shipping | Отгрузка клиенту: даты отправки и таможенная декларация (ГТД / GTD). |
| УПД | UPD | Универсальный передаточный документ — закрывающий учётный документ. Один УПД может покрывать несколько заявок. |
| Реквизиты | Customer account | Карточка юридического лица клиента — название компании, ИНН, банковские реквизиты, адреса. Отображается в счёте и УПД. Сущность `RequestCustomerAccount`. |
| Менеджер | Manager | Продавец компании, ведущий заявку. Кабинет показывает его имя, email, аватар и телефон компании. |

## 4. Роли, доступ, аутентификация

- **Аутентификация и авторизация — вне рамок этого документа.** Клиент уже
  приходит с HTTP-only cookie, которую принимает бэкенд. Кабинет не управляет
  cookie и не реализует форму входа.

  Если бэкенд возвращает 401, кабинет перенаправляет на страницу входа
  `https://id.snab.work`. После успешного входа кабинет возвращает обратно на
  исходную страницу.

  Если бэкенд возвращает 403, кабинет перенаправляет на страницу отказа в
  доступе `/access-denied`. Страница отказа в доступе показывает сообщение о
  том, что клиенту не разрешено выполнить запрошенное действие.

- Клиент — это `User`, связанный один-к-одному с `Customer`. Кабинет работает в
  контексте одного клиента.

- Сервер определяет клиента из сессионной cookie. Запрос кабинета (раздел 8) не
  принимает ID клиента — клиент берётся из сессии.

- Бэкенд возвращает только данные этого клиента, уже отфильтрованные до полей,
  которые клиенту разрешено видеть.

- **Истечение сессии.** Когда запрос завершается ошибкой из-за отсутствующей или
  истёкшей сессии (HTTP 401 или ошибка аутентификации GraphQL), кабинет
  перенаправляет на внешний URL входа, принадлежащий окружающей системе —
  собственной формы входа у него нет. Точный URL должен задаваться через
  переменные окружения.

## 6. Жизненный цикл заявки

Центральный документ — **Заявка**. Каждый другой документ создаётся в течение
жизни заявки и всегда принадлежит ей.

```
        ┌──────────┐
        │  Заявка  │  клиент описывает потребность: товары, желаемая цена
        └────┬─────┘
             │  менеджер обрабатывает заявку
             ▼
        ┌──────────┐
        │    КП    │  коммерческое предложение, отправленное клиенту
        └────┬─────┘  (структура = проекция счёта)
             │  клиент соглашается
             ▼
        ┌──────────┐
        │   Счёт   │  выставлен клиенту; клиент вносит платежи
        └────┬─────┘
             │  оплата запускает исполнение
             ▼
        ┌──────────┐
        │  Заказ   │  компания закупает товар; клиент видит
        └────┬─────┘  даты заказа и статус груза
             │  формируется груз
             ▼
        ┌──────────┐
        │ Доставка │  отгрузки: даты отправки, таможенная декларация (ГТД)
        └────┬─────┘
             │  товар передан клиенту
             ▼
        ┌──────────┐
        │   УПД    │  закрывающий учётный документ. Может покрывать несколько заявок.
        └──────────┘

Реквизиты (RequestCustomerAccount) — карточка юридического лица клиента;
принадлежат заявке и отображаются в счёте и УПД.
```

Как документы соответствуют сущностям бэкенда:

| Документ | Сущность | Связь с заявкой |
|---|---|---|
| Заявка | `Request` | — |
| КП | `Proposal` (новая — см. раздел 8) | один-ко-многим от заявки |
| Счёт | `Invoice` + `InvoiceProduct` + `InvoicePayment` | один-ко-многим от заявки |
| Заказ | `Order` | один-ко-многим от заявки |
| Доставка | `OrderShipping` + `OrderShippingGroup` + `OrderShippingProduct` | один-ко-многим внутри заказа |
| УПД | `Upd` + `UpdProduct` + `UpdRequest` | многие-ко-многим с заявками |
| Реквизиты | `RequestCustomerAccount` | один-к-одному с заявкой |
| Товары | `Product` (позиции) и `productsRequested` (что запросил клиент) | от заявки |

## 8. Данные

Данные кабинета в основном строятся на запросе сущностей `Request` и `Customer`.
Главные сущности — авторизованный клиент и его заявки, с каждым документом,
вложенным в заявку.

Запрос может быть составлен из двух следующих фрагментов в папке `queries/`:

- `queries/CustomerAggregate.gql` — клиент.
- `queries/RequestAggregate.gql` — одна заявка с её КП, счетами, заказами,
  отгрузками и УПД.

Например, их можно собрать в единый запрос:

```graphql
query CustomerCabinet {
  customer { ...CustomerAggregate }
  requests { ...RequestAggregate }
}
```

Клиент берётся из сессии, поэтому запрос не принимает аргументов. Типы полей для
каждой сущности — в папке `schema/`.

`productID` и `requestID` внутри `shippingProducts`, `updProducts` и
`updRequests` — внешние ключи; сохраняйте их для соединений на стороне клиента
(один УПД может покрывать несколько заявок клиента).

### 8.1. Будет добавлено в схему бэкенда

`RequestAggregate` уже использует три вещи, которые бэкенд пока не предоставляет.
Их нужно добавить в схему GraphQL:

```graphql

extend type Request {
  "Commercial proposals (КП) for this request."
  proposals: [Proposal!]!
}

"A commercial proposal (КП). A pre-payment projection of an invoice."
type Proposal {
  id: ID!
  created: Int!
  changed: Int!
  title: String
  actuality: String
  shippingTime: String
  prepaid: Float
  companyDetails: String
  proposalProducts: [ProposalProduct!]!
}

"A line item of a proposal — same shape as an invoice line item."
type ProposalProduct {
  id: ID!
  lineNum: Int
  title: String
  qty: Float
  currency: String
  vat: Int
  unit: Float
  unitNoVat: Float
  line: Float
  lineNoVat: Float
}
```

КП повторяет структуру счёта, но не имеет полей оплаты (`paid`, `paidInput`,
`firstPaymentCreated`, `canceled`) — до оплаты они не имеют смысла. У заявки
может ещё не быть КП, поэтому кабинет должен обрабатывать пустой список
`proposals`.

### 8.2. Пример ответа

Полный пример полезной нагрузки для запроса `CustomerCabinet` — один клиент с
двумя заявками, одна полностью заполненная и одна минимальная — находится в
[`examples/cabinet-response.json`](../examples/cabinet-response.json). Используйте
его для построения UI, пока резолвер бэкенда ещё не существует.

## 9. Расчёты: счёт и КП

### Позиции

Каждая позиция счёта или КП отображается из следующих полей:

- `title` — описание товара.
- `qty` — количество.
- `unit` — цена за единицу, в `currency` позиции. Показывайте её только когда
  `qty > 1`; при `qty == 1` цена за единицу равна итогу строки и не показывается.
- `line` — итог строки, в `currency` позиции. Показывается всегда.

Форматируйте каждое денежное значение в `currency` позиции (`RUB` или `EUR`).

### НДС

`unit` и `line` — цены с НДС. `unitNoVat` и `lineNoVat` — те же цены без НДС.
`vat` — ставка НДС в процентах (например, `20`). По умолчанию показывайте
значения с НДС; детализацию без НДС следует размещать за элементом «показать
больше».

### Итог счёта

Сохранённого итога нет — вычисляйте его на клиенте:

```
total = Σ line   over line items where active == true
```

Суммируйте только позиции, где `active == true`; неактивные позиции не
учитываются. Форматируйте итог в валюте счёта.

Примеры вспомогательных функций для счёта — в файле `examples/invoice.ts`.

### Инвариант валюты

Все позиции в пределах одного счёта имеют одну и ту же `currency`. Итог и его
форматирование опираются на это.

### Платежи

Каждый `InvoicePayment` содержит:

- `transferDate` — дата платежа.
- `transferNumber` — номер платёжного поручения.
- `paidPercent` — доля этого платежа в итоге счёта, в процентах.
- `paidInput` — оплаченная сумма, всегда в RUB.

Общий прогресс оплаты, агрегированный по всем платежам, — это `invoice.paid`
(процент, 0–100).

### Счета в EUR — тройка сумм

Карточка заявки или страница деталей заявки показывает тройку сумм
«выставленная сумма / оплаченная сумма / процент оплаты» (раздел 11). Для
счёта в RUB это просто. У счёта в EUR есть несовпадение валют: его итог в EUR,
а платежи (`paidInput`) всегда в RUB. Тройка сумм не должна смешивать валюты:

- **Выставленная сумма:** вычисленный итог, в валюте счёта (EUR).
- **Оплаченная сумма:** оплаченная сумма, в RUB.
- **Процент оплаты:** `invoice.paid` в процентах (например, «оплачено 50%»).

Все позиции счёта имеют одну и ту же валюту.

## 11. Разделы кабинета — Этап 1

Согласно разделу 2, кабинет — это три экрана: лента заявок (главная), детали
заявки и профиль. Архив (11.4) — это фильтр на ленте, а не четвёртый экран.
Конкретный UX — на усмотрение разработчика.

### 11.1. Главная — лента заявок

Главная — это дашборд клиента.

- Клиент видит все свои активные заявки в одном месте.
- По каждой заявке клиент может с одного взгляда понять:
  - что было заказано — краткое описание товаров;
  - где находится заявка — её позицию в 4-шаговом трекере статусов
    (раздел 7): обработка → КП → доставка → готово;
  - деньги — выставленная сумма, оплаченная сумма и процент оплаты;
  - что будет дальше (например, «ожидается оплата», «товар в пути»).
- Клиент может открыть любую заявку с главной, чтобы увидеть её целиком
  (раздел 11.2).
- Клиент может переключиться на завершённые и отменённые заявки (раздел 11.4).

Для большинства клиентов главная — это весь кабинет: у них всего 2–3 активные
заявки.

### 11.2. Детали заявки

Страница заявки — это полное отображение всего, что кабинет знает об одной
заявке, на одной странице или в виде модального окна / выдвижной панели.

- Клиент видит ход заявки как понятную последовательность шагов —
  создана → КП → счёт → заказ → доставка → завершена — показывающую, где заявка
  сейчас, что уже сделано и что ещё впереди.
- Клиент видит каждый документ заявки: КП, счёт, заказ и его доставку, УПД и
  реквизиты.
- Клиент видит денежную сводку по заявке и может раскрыть её до полной
  детализации по позициям.

Схематичный макет — см. Приложение A.

### 11.3. Профиль

Личные данные клиента и реквизиты юридического лица, взятые из самой последней
заявки. Этап 1: только для чтения.

### 11.4. Архив

Архив — это **фильтр на ленте заявок**, который показывает отменённые и
завершённые заявки. Он в стороне от основного пути и свёрнут по умолчанию.

## 12. Этапы 2 и 3 (обзор)

Не входит в Этап 1; приведено здесь для контекста.

**Этап 2 — мутации:**

- Создание заявки. Форма — это набор товарных строк: бренд (выбор из списка),
  категория (выбор из списка), модель / SKU (свободный текст), количество
  (свободный текст); опционально итоговая «цена, которую клиент готов
  заплатить» (`customerDesiredPrice`).
- Редактирование телефонов клиента.
- Подготовка процесса работы с КП.

**Этап 3 — возможности:**

- Принятие или отклонение КП; подтверждение счёта.
- Редактирование реквизитов — с проверкой на бэкенде, что реквизиты не
  используются заявкой, которая уже перешла в заказ.
- Чат клиент ↔ менеджер — со своими отдельными сущностями.
- Напоминания о звонках — уведомление клиента о предстоящем звонке.
- Опросники (квиз).
- Событийные уведомления (новый счёт, смена статуса, отгрузка).

**После Этапа 3:** интернационализация (i18n).

## 13. Нефункциональные требования

- **Язык интерфейса — только русский.** Интернационализация — после Этапа 3.
- **Mobile-first.** Клиенты заходят в основном с телефонов: узкие блоки,
  одноколоночная вертикальная вёрстка.
- **Простота.** Весь кабинет клиента умещается в один запрос `CustomerCabinet`,
  и набор данных невелик. Стандартный GraphQL-клиент с нормализованным кэшем —
  это весь слой данных.
- **Визуальный язык** согласуется с заказчиком отдельно; в этом документе он не
  фиксируется.

## 14. Замечания по бэкенду, которые нужно знать

То, что не очевидно из одной только схемы:

- **Временные метки** — это **секунды** Unix-эпохи (целые числа), а не строки
  ISO и не миллисекунды. Это касается полей `created`, `changed`, `date`,
  `postDate` и подобных.
- **Деньги.** `unit` и `line` включают НДС; `unitNoVat` и `lineNoVat` — без
  него; `vat` — целое число процентов (например, `20`). `paidInput` всегда в
  RUB, независимо от валюты счёта. Значения `currency` — это ISO-коды в верхнем
  регистре (`RUB`, `EUR`).
- **Одна валюта на счёт** — все позиции счёта имеют одну `currency` (раздел 9).
- **Специальные значения.** Временная метка `0` означает «нет даты»;
  `customerDesiredPrice`, равная `0`, означает «не указана»; `customerAccount`
  может быть `null` (до выставления счёта); отсутствующие строковые поля
  приходят как пустые строки.
- **Точка доступа GraphQL.** URL точки доступа берётся из переменной окружения,
  задаваемой собственной конфигурацией бэкенда кабинета.
